# 内容 Skills 架构与调用顺序

## 这份文档解决什么问题

这份文档用来定义当前这套内容型 skills 的分层关系、职责边界和调用顺序。

目标不是再做一个“大而全”的总控 skill，而是把原来的复杂链路拆成几层可复用能力：

- 账号基础层
- 选题审查层
- 母稿生产层
- 平台改写层
- 发布执行层

这样后面无论新增公众号、小红书图文、短视频脚本、视频制作，前面的公共能力都不需要重做。

---

## 一、当前拆分思路

原来的 `social-media-context-sms` 更像一个大入口，容易把这些事情混在一起：

- 账号定位
- 选题判断
- 风险审查
- 正式写作
- 平台改写

现在拆开以后，链路更清楚：

1. `account-context`
2. `scan-score-topic`
3. `article-write`
4. 各平台改写 skill
5. 发布或制作 skill

这个拆法更适合你现在的账号，因为你本来就是：

- 一个长期稳定定位的账号
- 会持续做月度选题
- 会把一篇母稿复用到多平台
- 后面还会继续补视频和平台执行链路

---

## 二、每个 skill 的职责

### 1. `account-context`

这是账号基础档案层。

它只负责维护长期相对稳定的信息，不负责选题，不负责正式写作。

它要解决的是：

- 这个账号是谁
- 写给谁
- 长期讲什么
- 用什么语气说
- 哪些表达不能说
- 哪些内容不做

它的典型产物应该是统一账号档案，比如：

- `.agents/account-context.md`

对你这个账号来说，这一层沉淀的就是：

- `土豆学AI / AI趣创社`
- 普通创作者、个体经营者、轻创业人群
- 主线只讲 `AI 工作流 / 内容提效 / 轻变现`
- 不做纯资讯号，不做纯工具盘点号

### 2. `scan-score-topic`

这是写作前的选题闸门层。

它不写正文，只判断：

- 这个题值不值得做
- 适不适合这个账号做
- 有没有风险
- 容不容易写偏
- 后续应该怎么写，边界在哪里

它产出的不是文章，而是“选题规则包 / 写作指导包”。

这一步的本质是：

把“一个想法”变成“一个可安全执行、可继续写的题”。

### 3. `article-write`

这是母稿层。

它不负责重新选题，也不负责平台化。

它只做一件事：

基于：

- `account-context`
- `scan-score-topic` 产出的规则包

写出一篇可复用的中文母稿。

这个母稿不是公众号成品，也不是小红书成品，而是平台共用的中间层。

对你来说，它更像：

- 观点已经立住
- 结构已经清楚
- 案例和动作已经有了
- 后面各平台只要继续翻译成适合的形式

### 4. 平台改写 skills

这是平台执行前的适配层。

以后你应该把它继续拆成：

- 公众号长文改写 skill
- 小红书图文改写 skill
- 短视频口播脚本改写 skill
- 视频分镜拆解 skill

这一层的原则是：

只改平台表达，不重做账号定位，不重做选题，不重写母稿逻辑。

也就是说，它们吃的是同一篇母稿。

### 5. 发布 / 制作 skills

这是最末端的执行层。

比如：

- 微信公众号推送
- 小红书发布准备
- 视频制作
- 配图生成
- 封面制作

这一层只负责交付，不再碰前面的定位、选题和母稿核心判断。

---

## 三、这些 skill 的上下游关系

可以把它理解成一条标准内容流水线：

```text
account-context
  -> scan-score-topic
  -> article-write
  -> 平台改写 skill
  -> 发布/制作 skill
```

如果翻成更直白的话：

- `account-context` 决定“你是谁”
- `scan-score-topic` 决定“你写什么”
- `article-write` 决定“这篇怎么写成母稿”
- 平台改写层决定“这篇怎么变成公众号 / 小红书 / 视频”
- 发布执行层决定“怎么真正落地出去”

## 四、你这个账号的推荐调用顺序

### 场景 A：账号定位没变，做日常内容

这是你现在最常见的情况。

调用顺序应该是：

1. `scan-score-topic`
2. `article-write`
3. 平台改写 skill
4. 发布 / 制作 skill

这里不需要每次都重新跑 `account-context`，因为账号底层没变。

### 场景 B：账号定位发生变化

比如：

- 主线变化
- 受众变化
- 语气变化
- 商业承接变化

这时应该先更新：

1. `account-context`
2. `scan-score-topic`
3. `article-write`
4. 平台改写

### 场景 C：你要做新的平台子流程

比如后面新增：

- 小红书图文链路
- 视频脚本链路
- 视频制作链路

正确做法不是把前面的逻辑再复制一遍，而是直接把新 skill 挂到 `article-write` 后面：

```text
account-context
  -> scan-score-topic
  -> article-write
  -> 新平台 skill
```

## 五、你后续新增 skills 时的边界建议

### 1. 公众号 skill

只负责：

- 把母稿改成公众号长文
- 做公众号分节、过渡、排版前结构

不负责：

- 账号定位
- 选题判断
- 热点审查

### 2. 小红书图文 skill

只负责：

- 把母稿改成小红书图文表达
- 拆封面标题、页间节奏、卡片结构

不负责：

- 重新定题
- 重新改账号定位
- 重写整套逻辑

### 3. 短视频脚本 skill

只负责：

- 母稿 -> 口播脚本
- 母稿 -> 镜头节奏
- 母稿 -> 分镜大纲

不负责：

- 选题重做
- 平台发布
- 视频剪辑成片

### 4. 视频制作 skill

只负责：

- 口播稿 / 分镜 -> 视觉方案
- 画面、配图、配音、剪辑、字幕

不负责：

- 账号定位
- 选题审题
- 母稿逻辑

### 5. 配图 / 封面 skill

只负责：

- 给现有内容生成配图
- 维持统一 IP 和视觉系统

不负责：

- 改写内容主线
- 替代正文判断

## 六、为什么这样拆最适合你

因为你不是做“一次性发一篇”的账号，而是在搭一个能长期复用的内容系统。

你这套账号有几个特点：

1. 账号定位长期稳定
2. 内容不是靠追热点吃饭
3. 同一主题会在多平台复用
4. 内容后面要承接免费资料、私域、低客单、训练营

这就意味着：

- 账号定位应该沉淀一次，长期复用
- 选题判断应该先过闸门
- 正文应该先形成母稿
- 平台只是后端分发形式

如果不这么拆，后面最容易出现的问题就是：

- 公众号一套逻辑
- 小红书另一套逻辑
- 视频又一套逻辑
- 最后整个账号主张越做越散

## 七、推荐的标准工作流

### 日常工作流

```text
月度选题需求
  -> scan-score-topic
  -> 月度选题文档
  -> article-write
  -> 文章总览/母稿
  -> 平台改写 skill
  -> 平台草稿目录
  -> 发布/制作 skill
  -> 复盘
```

### 资产沉淀工作流

```text
账号变化
  -> account-context
  -> .agents/account-context.md

高频选题模式
  -> 月度选题沉淀

高频表达模板
  -> 模板库沉淀

高频承接动作
  -> 转化动作沉淀
```

## 八、和旧 `social-media-context-sms` 的关系

旧的 `social-media-context-sms` 以前像一个“大总管”。

现在它更适合退成“迁移提示器”，只负责告诉调用者：

- 别再把所有事塞给一个 skill
- 该用 `account-context` 的时候就用 `account-context`
- 该用 `scan-score-topic` 的时候就用 `scan-score-topic`
- 该写母稿的时候就用 `article-write`

也就是说，它不该再承担核心生产任务。

---

## 九、你现在最适合的落地方式

对你当前工作区，最合理的方式是：

1. 把 `account-context` 作为公共底座
2. 把 `scan-score-topic` 作为月度选题入口
3. 把 `article-write` 作为母稿入口
4. 后面继续补：
   - 公众号改写 skill
   - 小红书图文改写 skill
   - 短视频脚本 skill
   - 视频制作 skill
5. 所有末端 skill 都不要重复碰前面的公共层

---

## 十、最终原则

一句话总结：

不是每个平台都各自生产一遍内容，而是先有统一账号底座，再有统一选题闸门，再有统一母稿，最后再分平台执行。

你后面所有新增 skill，都应该问自己一句：

**我这个 skill 是在补哪一层能力？**

如果答不上来，就说明边界还不够清楚。
