# 公众号图文生成

## 核心作用

这个 skill 只负责一件事：

把 `article-write` 产出的中文母稿，改造成一套可发布、可审核、可继续交给 `wechat-push-skill` 入草稿箱的公众号图文内容包。

## 强制参考关系

从现在开始，这个 skill 的内容与配图组织方式必须同时参考：

- `potato-illustrations`
- `redbook-gen`

其中：

- `potato-illustrations` 负责角色、风格、配图表达、土豆崽一致性
- `redbook-gen` 负责标题思路、封面方案结构、图文拆解方式、呈现方式、审核标准的组织方法

也就是说，`wechat-gen` 不是另起一套松散方案，而是：

1. 读 `potato-illustrations`
2. 读 `redbook-gen`
3. 继承小红书那套“标题库、封面方案、拆解、审核”的高可读结构
4. 再改造成更适合公众号文章阅读的版本

## 强制图像依赖

- 公众号正文配图必须优先且默认使用 `potato-illustrations`
- 公众号封面视觉也必须以 `potato-illustrations` 的土豆崽角色与风格体系为默认起点
- 不允许跳过 `potato-illustrations` 直接写一套泛化图片 prompt
- 不允许把土豆崽降级成“可选角色”
- 不允许把公众号配图做成通用商业插画风
- 不允许在 `wechat-gen` 内直接调用任何其他生图工具、模型或接口
- 如果需要生成图片，唯一入口是 `potato-illustrations`

## 必读依赖

开始前按顺序读取：

1. `$article-write`
2. `$potato-illustrations`
3. `$redbook-gen`
4. `$wechat-push-skill`

再读取本 skill 内部文件：

- `references/workflow.md`
- `references/output-spec.md`
- `references/review-standard.md`
- `examples/testcase-20260701.md`

## 输出结构

输出目录：

```text
06_运营中心/<年份>/<YYYY-MM>/公众号/<YYYY-MM-DD>/<YYYYMMDD_公众号_<标题短名>/
```

必须包含：

```text
发布稿.md
草稿箱正文.txt
配图方案.md
封面方案.md
metadata.json
审核记录.md
prompts/
images/
```

## 这次开始必须新增的结构文件

除了原有文件，`wechat-gen` 还必须额外产出：

```text
正文结构与配图映射.md
```

这个文件不是可选说明，而是后续 `wechat-push-skill` 入草稿时的执行清单。

## 标题层级要求

公众号不是一整篇只有正文。

从现在开始，`wechat-gen` 生成公众号稿时，必须先把正文改造成清晰层级：

- 一级标题：文章主标题，只出现 1 次
- 二级标题：主节标题，例如 `一、为什么很多人用了 AI 还是没提效`
- 三级标题：提示型小标题，例如 `先看结论：`、`真正的问题是：`

要求：

- 不能整篇只有正文长段落
- 不能只有一个总标题，没有中间层级
- 二级标题必须承担结构分段作用
- 三级标题必须承担阅读节奏作用
- 标题层级要写进 `发布稿.md`、`草稿箱正文.txt` 和 `正文结构与配图映射.md`

## 呈现方式要求

这一部分强烈借鉴 `redbook-gen`：

### 1. 标题

不要只给一个标题。至少给：

- 主标题
- 副标题
- 备选短标题 3 个
- 点击理由

### 2. 封面方案

封面方案必须像 `redbook-gen` 一样写清：

- 主标题
- 副标题
- 备选短标题
- 点击理由
- 画面主视觉
- 角色表情
- 配色
- 标题叠加建议
- 实际交付说明

### 3. 配图方案

配图方案必须参考 `redbook-gen` 的拆解方式，不再只写一句话。

每张正文图至少写清：

- 图号
- 对应段落
- 本图观点
- 图中可见短文案
- 画面内容
- 土豆崽动作与表情
- 背景与道具
- 配色
- 构图方向
- Prompt 路径
- 审核要点

另外必须补一层执行信息：

- 插入位置编号
- 插入在第几个二级标题后
- 插入在第几个段落后
- 推送时是否允许挪动

也就是说，`配图方案.md` 是图片创作说明，`正文结构与配图映射.md` 是推送执行说明，这两个不能混为一谈。

### 4. 审核记录

审核记录也要参考 `redbook-gen` 的呈现方式：

- 结论
- 检查项表格
- 需要重做

## 公众号适配规则

虽然参考 `redbook-gen` 的组织方式，但公众号不是小红书。

所以必须额外满足：

- 文章为主，配图为辅
- 正文图统一横版优先
- 单张图只解释一个判断
- 配图不能有“小红书卡页感”
- 不能像海报
- 不能空
- 封面可以更有冲突，但正文图必须克制

## 封面规则

- 封面必须单独设计
- 封面必须有标题成品，不允许只交无字底图
- 公众号封面尺寸必须按平台双场景交付，不能只做一张竖版海报图
- 必须同时满足：
  - 主封面卡：`2.35:1` 横向比例，用于后台主文章卡、列表卡，默认推荐直接输出 `2820 x 1200`
  - 方形封面卡：`1:1` 比例，用于转发卡、公众号主页等场景，默认推荐直接输出 `1600 x 1600`
- 不允许只交 `3:4` 或其他竖版图后让后台自动裁切
- 不允许把一张竖版标题海报直接上传为公众号主封面
- 如果需要保留竖版视觉稿，只能作为设计中间稿或补充稿，不能作为 `metadata.json` 的 `cover_path`
- 标题必须本地叠加，不能把中文大标题直接交给模型去生成
- 标题必须使用稳定中文字体本地渲染
- 标题文字来源必须是 UTF-8 文本文件，不允许 shell 里临时拼中文大段
- 封面标题区必须先定义安全区，再叠字，不能先出字后碰运气
- 标题区右侧不得侵入高密度道具区、角色区、杂线区
- 长标题必须先拆成 2 到 3 行，再决定字号，不能硬撑一整行
- 标题框宽度宁可保守，也不要为了“铺满”去压到主体或噪点区
- 标题块必须做缩略图检查：缩小后仍可读，且不显得头重脚轻
- 封面图默认按公众号封面逻辑单独输出，不得把正文横图放大后冒充封面
- 封面图必须优先使用公众号主卡适配横图作为默认封面成品，不得与正文图尺寸混用
- 主封面标题安全区必须按横向卡片重新审视，不能沿用竖版海报的居中排布
- 方形封面必须单独检查居中裁切效果，不能简单从主封面粗暴拉伸
- 封面成品默认交付：
  - `01_公众号封面_标题版_2.35比1.png`
  - `01_公众号封面_标题版_1比1.png`
  - `01_公众号封面_底图_2.35比1.jpeg`
  - `01_公众号封面_底图_1比1.jpeg`
- `metadata.json` 里的 `cover_path` 默认必须指向 `01_公众号封面_标题版_2.35比1.png`
- 微信公众号封面审核时，先看后台列表卡缩略图，再看编辑页封面预览；任一场景出现明显挤压、裁断、标题吃字，直接打回

## 文字渲染硬规则

这次开始，`wechat-gen` 内部必须明确：

- 所有封面和图片上的中文标题、短标、说明，优先本地渲染
- 不能依赖模型直接画中文大标题
- 本地叠字必须使用稳定中文字体
- 本地叠字时，文本源文件必须是 UTF-8
- 本地叠字前必须先判断底图的高密度区域和标题安全区
- 本地叠字时必须做最大行宽限制，超宽就拆行或降字号
- 禁止把长标题直接铺满整个标题框
- 如果生成结果出现 `?`、乱码、错字、缺字，直接打回

## 正文结构与配图映射要求

`正文结构与配图映射.md` 至少要写成下面这种粒度：

```markdown
# 正文结构与配图映射

## 文章层级

- 一级标题：不是你不会用 AI，而是你一直没有自己的第一条工作流
- 二级标题 1：一、为什么很多人用了 AI 还是没提效
- 二级标题 2：二、真正缺的不是工具，而是第一条工作流
- 二级标题 3：三、普通人该怎么开始搭第一条工作流

## 配图映射

### 图 01 封面
- 文件：images/01_公众号封面_标题版.png
- 用途：封面
- 不进入正文

### 图 02 正文图
- 文件：images/02_正文配图_点状使用_redo.jpeg
- 插在：二级标题 1 下，第 3 段后
- 作用：解释“点状使用”

### 图 03 正文图
- 文件：images/03_正文配图_工作流路径_redo.jpeg
- 插在：二级标题 2 下，第 2 段后
- 作用：解释“工作流路径”

### 图 04 正文图
- 文件：images/04_正文配图_三个问题_redo.jpeg
- 插在：二级标题 3 下，第 2 段后
- 作用：解释“三个问题”
```

硬规则：

- 封面图只能用于封面
- 正文图必须逐张绑定到明确段落位置
- `wechat-push-skill` 后续必须按这个映射推送，不允许自行猜位置
- 如果缺这个文件，视为稿包不合格

## 一票打回

出现任一项，直接打回：

- 没读 `potato-illustrations` 就开始出图
- 没参考 `redbook-gen` 的结构就直接散写
- 封面没有标题成品
- 封面标题乱码、问号、错字、缺字
- 封面主交付仍是竖版图，未补齐 `2.35:1` 横封面
- 未交付 `1:1` 方形封面适配稿
- `metadata.json` 的 `cover_path` 仍指向竖版图或非主卡比例图
- 封面标题冲进杂物区、箭头区、角色区
- 标题框过宽、过高，导致画面头重脚轻
- 长标题未拆行，硬撑导致穿模或不协调
- 没有生成 `正文结构与配图映射.md`
- 没有二级标题和三级标题结构
- 正文图没有明确插入位置映射
- 把封面图混进正文，或拿正文图冒充封面
- 角色漂移明显
- 画面像泛化商插
- 正文图像小红书卡页
- 画面太空
- 文字压脸、穿模、贴边

## 防乱码规则

本 skill 全部中文文档固定使用 UTF-8。

一劳永逸的规则：

1. 所有 `.md`、`.json`、`.txt` 均使用 UTF-8 保存
2. 中文长文不要在终端里拼接写入
3. 出现乱码时整文件重写，不在乱码文本上修补
4. 图片中的中文标题与短标优先本地渲染
5. 本地叠字时使用稳定中文字体
6. 不允许把 shell 输出里的乱码文本再拿去做叠字源
