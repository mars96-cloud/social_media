# 小红书图文生成

## 核心目的

这个 skill 承接 `article-write` 的中文母稿，专门在本地生成小红书图文内容包。

它负责本地完整交付，不是只写 Markdown：

- 能吸引点击的标题库
- 3:4 竖版封面方案和封面图
- 7 到 9 页图文卡片拆解
- 土豆崽 IP 插图 prompt 和正式图片
- 小红书发布正文
- 标签与结尾引导
- 本地交付前审核报告

完成定义：

- 只生成 `选题转译.md`、`标题库.md`、`封面方案.md`、`图文卡片拆解.md`、`发布稿.md`、`审核报告.md`、`prompts/`，不算完成。
- 必须继续基于 `prompts/` 和卡片拆解，把封面和内页交给 `potato-illustrations` 生成正式图片，并保存到 `images/`。
- 每篇默认至少交付 `01_封面.png`、`01_封面_标题版.png`、`02_痛点.png` 到 `08_收束.png`。
- 图片生成后必须复检文字、穿模、贴边、角色一致性和画面密度，再更新 `审核报告.md`。
- 只有当 Markdown、prompts、正式 images 和审核报告都齐全时，`redbook-gen` 才算完成。

默认插图风格和所有图片生成任务都必须使用 `potato-illustrations`。`redbook-gen` 只负责把母稿拆成小红书内容包、配图需求、卡片说明和审核标准；不直接调用任何生图工具、模型或接口。

需要生成图片时，唯一入口是 `potato-illustrations`。`redbook-gen` 不能绕过它写独立生图流程，不能绕过它调用其他生图能力，不能把本地渲染预览冒充正式图片。

注意：`redbook-gen` 可以组织和调度完整交付流程，但实际图片生成必须由 `potato-illustrations` 执行。不得因为批量生成、赶进度、网络失败或已有本地预览图，就停在 Markdown 阶段并声称任务完成。

## 硬边界

本 skill 不负责、也不允许执行任何小红书线上动作。

一律禁止：

- 调用 `xhs-search`
- 调用 `xhs-explore`
- 调用 `xhs-login`
- 调用 `xhs-interact`
- 调用任何小红书 MCP、API、浏览、搜索、详情、评论、主页、推荐流能力
- 主动搜索、刷新、查看、抓取、爬取任何线上小红书数据
- 引导扫码登录小红书
- 自动发布、点赞、收藏、评论、回复、切号

`xhs-search` 和 `xhs-explore` 只能作为边界参考：它们说明哪些能力属于线上搜索、浏览、详情和评论读取能力，因此不能被本 skill 调用。不要把它们的执行流程合并进本 skill。

如果用户提供小红书截图、笔记文本、链接摘要、历史本地分析文件，可以只把这些用户提供或工程内已有材料当作参考。没有参考材料时，直接基于母稿、账号定位和本 skill 规则生成，并在 `审核报告.md` 写明：

> 未调用小红书 API，未使用实时小红书数据，本稿基于母稿和账号定位生成。

## 必读依赖

按任务读取：

- `article-write`：确认母稿只是共同故事模板，不是小红书成品。
- `potato-illustrations`：唯一图片入口，负责图像风格、土豆崽 IP、角色一致性、图片生成流程和图片质检。不允许只画一个泛化土豆角色。
- `xhs-content-plan`：只参考本地策划方法、标题分类、标签策略和审核思路，不调用任何小红书接口。

## 平台规格

小红书图文卡片默认使用竖版：

- 默认比例：`3:4`
- 推荐尺寸：`1242 x 1660`
- 兼容尺寸：`1080 x 1440`
- 如 `potato-illustrations` 的图片产出尺寸更高，最终小红书成品仍必须裁切或排版为 `3:4`
- 封面和内页统一同一比例，不混用横版、方图和竖图
- 重要文字放在安全区内，离四边至少 80 px
- 标题、人物脸、关键道具不能贴边、被裁切或互相遮挡

说明：当前未查到稳定可引用的官方公开规格页。以上按小红书图文实操常用规格固化；发布前如平台规则变化，应复核。

## 已验证的稳定路线

当前工作区已经验证过一条可复用路线：`2026-07-01` 小红书稿包。

以后做同类土豆崽小红书图文时，优先沿用这套经验：

- 每篇默认 8 页：封面、痛点、判断、核心、代入/自测、方法、场景、收束。
- 图片比例统一 `3:4`。
- 封面使用“底图 + 本地叠字”，不要让模型直接生成大标题。
- 封面主标题要大、短、居中偏上，优先 8 到 18 个汉字。
- 副标题只能短句，放在标题下方安全区，不要压到角色脸和身体。
- 装饰线只能作为局部强调线，不能横穿画面、角色、脸部、手部或关键道具。
- 正文页图中文字控制在 10 到 35 个汉字，最多 2 句。
- 土豆崽必须承担核心动作，不能只是旁边站着。
- 如果封面第一次叠字压脸、贴边或穿模，必须立即重导出标题版，不要把问题图标记为通过。
- 本地图文卡渲染时，主体视觉区要足够饱满：土豆崽、道具、路径或对比元素应占据画面中部主要空间，不能只有一个小角色漂在大片空白里。
- 不要在图片底部放长解释句；解释放到 `发布稿.md`，图片只保留短标签和关键句。
- 箭头、强调线、路径线不能穿过中文、角色脸、手部或关键道具。
- 内页不要出现左上角“痛点”“判断”“核心”“收束”等页签。
- 不要把所有标题都用明显黑色框框起来；标题应像海报标题一样直接贴在上方安全区。
- 可以用轻微白色半透明底带保证可读性，但不要做成 PPT 表格标题框。

### 批量生成规则

当用户要求一次生成多篇时：

- 必须先读取 `文章总览/<YYYY-MM-DD>/` 下的母稿。
- 不能只凭月度选题表、选题大纲、标题或用户一句主题生成小红书包。
- 如果某一天没有文章总览母稿，必须停止该篇小红书生成，并提示先使用 `article-write` 生成母稿。
- 每篇单独建 `小红书/<YYYY-MM-DD>/<YYYYMMDD_小红书_<短标题>/`。
- 每篇都必须有 `选题转译.md`、`标题库.md`、`封面方案.md`、`图文卡片拆解.md`、`发布稿.md`、`审核报告.md`、`prompts/`、`images/`。
- 每篇都要单独写审核报告，不允许用一个总审核替代。
- 批量生成仍然禁止调用小红书 API/MCP；不得为了“学习热点”打开线上小红书。
- 如果实际图片只是本地渲染或本地叠字预览，要在审核报告中写清“预览图，不是正式图”，不能伪称已经通过 `potato-illustrations` 正式生成。

## 标准输入

- 若用户只有主题或选题大纲，必须先使用 `article-write` 生成文章总览母稿。

## 工作流程

### 1. 读母稿

从母稿里提炼：

- 一句话核心观点
- 读者痛点
- 反常识判断
- 最小行动建议
- 可复用金句
- 可画成图的隐喻
- 不适合进入小红书的长段落

不要平均切段。小红书不是公众号截图，必须重新组织成滑动节奏。

### 2. 本地参考表达

只能参考用户主动提供的材料或工程内已有材料，目的是辅助标题、结构、标签，不是为了登录、搜索、爬取或发布。

可参考：

- 用户粘贴的小红书笔记文本
- 用户提供的小红书截图
- 工程内已有历史小红书选题、发布稿、复盘或分析文件
- `xhs-content-plan` 的本地策划方法论

禁止：

- 调用小红书搜索、浏览、主页、评论等任何 API
- 主动爬取、抓取、刷新线上数据
- 为学习标题而登录小红书
- 编造“已搜索到的热门笔记”

如果没有可参考材料，就在 `选题转译.md` 和 `审核报告.md` 写明未使用实时小红书数据。

### 3. 起标题

生成 `标题库.md`，至少包含：

- 强痛点标题 5 个
- 反常识标题 5 个
- 场景代入标题 5 个
- 封面标题 3 个
- 最终推荐 1 个

标题标准：

- 一眼看出读者痛点
- 尽量有冲突或反差
- 不写空泛大词
- 不碰夸大收益、绝对承诺、恐吓式标题
- 不复刻热点笔记原题

封面标题控制在 8 到 18 个汉字。发布标题可以稍长，但不要像公众号标题。

### 4. 做封面

封面必须单独设计，不能直接拿第 1 页正文卡凑数。

封面必须有：

- 大标题
- 明确冲突
- 土豆崽表情或动作
- 一个能让人停下来的视觉隐喻
- 干净标题区

封面优先做成“potato-illustrations 图片 + 本地叠字”：

- `potato-illustrations` 负责正式封面图片生成
- 标题后期本地叠加，避免模型中文乱码
- 标题不放在复杂框里，不压角色脸，不贴边
- 本地叠字必须复检：标题是否在安全区、副标题是否压脸、强调线是否穿过角色。
- 如果底图已经有可读中文短词，可以保留；如果模型生成乱码字，必须遮挡、裁掉或重出图。
- 本地渲染封面如果出现“主体小、空白大”的问题，必须放大角色和道具，并增加简短标签、路径或对比元素来补足视觉密度。

### 5. 拆图文卡

推荐 7 到 9 页：

1. 封面：痛点和冲突。
2. 痛点页：让读者觉得“这就是我”。
3. 判断页：给反常识结论。
4. 概念页：把抽象概念变成生活画面。
5. 自测页：用 2 到 3 个问题让读者代入。
6. 方法页：给最小起步动作。
7. 场景页：说明适合谁、怎么用。
8. 收束页：一句可记住的结论。
9. 结尾引导页：可选，用轻互动问题收尾，只写文案，不做账号互动。

每页只讲一个观点。每页卡片拆解必须写：

- 页码
- 这一页观点
- 卡片可见文案
- 画面内容
- 土豆崽动作、表情、位置
- 背景、道具、配色
- 图片比例和尺寸
- 图片风格
- 生图 prompt
- 审核要点

卡片文案要求：

- 正文页可见中文 10 到 35 个汉字，最多 2 句
- 不写“老师”“学生”等身份标签，除非用户明确要求
- 不把公众号段落塞进图片
- 图中文字必须短，解释放到 `发布稿.md`

### 6. 做插图

默认使用土豆崽 IP：

- 读取 `potato-illustrations`
- 土豆崽必须参与核心动作，不能只是站在旁边
- 同一组图里，脸型、颜色、眉毛、眼睛、嘴型要一致
- 风格是白底、手绘、少字、清爽、有一点怪，不做 PPT 信息图

需要图片时：

- 必须交给 `potato-illustrations`
- `redbook-gen` 只能提供页面目标、画面说明、可见短句、比例要求和审核要点
- 不允许在 `redbook-gen` 内直接调用其他生图工具或模型
- 不允许在 `redbook-gen` 内写“改走其他生图路线”的 fallback
- 每张图都必须按 `potato-illustrations` 的土豆崽 IP、角色锚点和质检规则执行
- 封面先交给 `potato-illustrations`，正文页再交给 `potato-illustrations`
- 中文标题和短句允许后期本地叠加，以避免模型中文乱码；但画面主体必须来自 `potato-illustrations`
- 本地图文卡只能作为阶段性预览，不能作为最终正式图片交付。用户要求“生成图 / 出图 / 正式图”时，必须走 `potato-illustrations`
- 批量生成时，也必须逐篇逐页完成图片，不允许只批量生成 Markdown 后停止
- 如果某一页图片失败，必须记录失败页码、原因和下一步，不允许跳过后继续标记整篇通过

批量小红书图片卡生成入口：

```powershell
python C:\Users\Administrator\.codex\skills\potato-illustrations\scripts\generate_redbook_images.py --days 2 --model doubao-seedream-5.0-lite 3 4
```

这是 `potato-illustrations` 的执行入口，不是 `redbook-gen` 直接调用底层生图工具。生成完成后必须检查每篇 `images/` 下是否有 9 张正式图片，并更新审核报告。

### 7. 写发布稿

`发布稿.md` 必须包含本地待发布文案：

- 标题备选
- 正文
- 标签
- 结尾引导

正文默认 120 到 220 字。风格要像真人分享，不像 AI 总结。

写法优先：

- 先说读者正在遇到的问题
- 再给一句判断
- 再给最小行动
- 最后用一句轻互动问题收尾

不要堆工具名，不要行业报告口吻，不要夸大结果。

### 8. 审核

完成后读取 `references/audit-checklist.md`，生成 `审核报告.md`。

结论只能是：

- `通过`
- `打回重改`
- `静态通过，图片待复检`

只有正式图片已经通过 `potato-illustrations` 生成、保存到 `images/`，并完成图片复检时，才允许写 `通过`。如果图片尚未生成，结论不能写 `通过`。

只要封面、标题、正文、插图、合规、线上边界任一关键项不合格，就打回重改。

## 一票否决

出现任一项，直接打回：

- 没有文章总览母稿就生成小红书图文包
- 只凭月度选题表、标题或大纲生成小红书正式包
- 调用了 `xhs-search`、`xhs-explore` 或任何小红书 API/MCP
- 主动搜索、浏览、抓取、爬取、登录或发布小红书
- 没读取 `potato-illustrations` 就开始出图
- 用户要求正式出图时，用本地渲染图冒充 `potato-illustrations` 正式图
- 绕过 `potato-illustrations` 调用其他生图工具、模型或接口
- 图片不是通过 `potato-illustrations` 生成或质检的结果
- `images/` 没有正式图片，只生成了 Markdown 和 prompts，却声称任务完成
- 土豆崽不是画面主角
- 角色漂移明显
- 图像像泛化商插，不像土豆崽正文解释图
- 文字乱码、穿模、压脸、贴边
- 封面不吸引人
- 画面空白过多或过满
- 文案 AI 味重

## 防乱码规则

所有文档必须使用简体中文和 UTF-8。

- 不要用 shell 拼接长中文。
- PowerShell 读取中文文件时使用 `-Encoding UTF8`。
- Python 写中文文件时显式使用 `encoding="utf-8"`。
- 发现乱码文件时，整文件重写，不做局部修补。
