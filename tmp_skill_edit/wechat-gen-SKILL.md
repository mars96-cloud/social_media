# 公众号图文生成

## 核心作用

这个 skill 只负责一件事：

把 `article-write` 产出的中文母稿，改造成一套可审核、可本地交付、可继续交给 `wechat-push-skill` 入草稿箱的公众号图文内容包。

它不负责：

- 重新发明配图体系
- 在本 skill 内直接生图
- 绕过土豆崽 IP 入口生成任何封面或正文图
- 用旧的通用配图、漫画风样张、编辑器底图、商业插画替代正式配图

## 强制依赖关系

开始执行前，必须按顺序完整读取：

1. `$article-write`
2. `$potato-illustrations`
3. `$redbook-gen`
4. `$wechat-push-skill`
5. 本 skill 自身的 `references/workflow.md`
6. 本 skill 自身的 `references/output-spec.md`
7. 本 skill 自身的 `references/review-standard.md`
8. 本 skill 自身的 `references/image-placement-template.md`
9. `examples/testcase-20260701.md`

如果其中任一关键 skill 或规则文件不可读，必须明确报错并停止，不允许自行脑补一套替代流程。

## 图片入口硬约束

从现在开始，`wechat-gen` 的所有图片相关工作都必须遵守下面的硬规则：

- 正文配图唯一入口是 `$potato-illustrations`
- 公众号封面底图唯一入口也是 `$potato-illustrations`
- `potato-illustrations` 真正出图时，唯一允许调用的底层生图能力是 `$ark-image-generation`
- `wechat-gen` 自身禁止直接调用任何图片模型、图片脚本、图片接口、图片助手或图片 MCP
- `wechat-gen` 自身禁止直接写任何通用生图 prompt 去替代 `potato-illustrations`
- `wechat-gen` 自身禁止复用任何旧的“公众号通用配图”“漫画风样张”“编辑器背景图”“无角色商业图”
- `wechat-gen` 自身禁止把现成旧图放大、裁切、换名后冒充新生成的土豆崽图片

也就是说，`wechat-gen` 只能：

- 规划文章结构
- 规划封面方案
- 规划正文配图需求
- 把正式图像生成任务明确交给 `potato-illustrations`
- 在收到正式图片结果后，组织本地稿包

它不能自己画图，也不能自己决定别的图像入口。

## 文字叠加规则

从现在开始，公众号正式图片的中文文字效果必须遵守下面规则：

- 模型只负责生成无字底图，中文标题、提炼、收束句全部由本地叠字脚本完成
- 禁止依赖模型直接画中文标题，不再接受“让模型顺手写标题”的方式
- 叠字必须优先让文字避开主体，不能盖住土豆崽的脸、手、核心道具、主要箭头和关键结构
- 叠字区域必须先做安全区判断，优先落在相对干净的留白处，不能死写固定坐标
- 正文图采用“一整块主信息面板”思路，不允许继续使用旧版那种几条白框硬切开的信息框
- 标题、要点、收束句必须有明确层级：标题最强，要点规整，收束句最轻
- 背景遮罩必须克制：高透明玻璃感或轻底，不允许厚白块把原图大面积盖死
- 要点必须分点对齐，行距、字距、边距统一，不允许视觉松散或拥挤
- 长标题必须自动缩放、自动换行，不能出现顶到边缘、压住要点、挤掉收束句
- 每次调整叠字逻辑后，必须做至少 5 张真实图片的本地冒烟测试，且至少覆盖封面、左侧信息面板、右侧信息面板、长标题、短标题这几类情况
- 只要文字面板盖住土豆崽脸、手、核心道具、关键箭头、主结构路径中的任一项，就视为失败，禁止交付
- 只要文字与主体发生明显压叠、穿过、错层、半遮挡，就视为失败，禁止交付
- 不能接受“虽然挡了一点但还能看”的灰度判断，遮挡检查按零容忍执行
- 如果当前 raw 图没有足够干净的文字安全区，脚本必须自动换布局；仍不满足时，必须重生 raw 图；再不满足时直接失败，不允许硬叠字
- 禁止把“强行加更厚的白底遮住主体”当作解决遮挡的方法
- 禁止为了避让而把标题缩到发虚、要点挤成难读小字
- 封面和正文图都必须通过同一套安全区审核，不允许只要求正文严格

## 冒烟测试规则

- 改动文字效果脚本后，必须至少输出 5 张本地冒烟图
- 冒烟图必须人工逐张检查以下项目：
  - 是否遮挡主体
  - 是否压住关键道具
  - 是否标题过大挤压正文
  - 是否要点规整且不拥挤
  - 是否收束句位置自然
  - 是否背景遮罩过重
- 冒烟不通过时，不允许先交 skill 文档，必须先修脚本再测

## 正式出图责任边界

- `wechat-gen` 负责定义每张图的标题、核心提炼、收束句和插图位置
- `potato-illustrations` 负责：
  - 调 Ark 生成无字 raw 图
  - 本地完成中文叠字
  - 输出正式成品图
- 正式效果不佳时，优先返工 `potato-illustrations` 的叠字脚本，不允许拿 prompt 乱补救

## 单一失败原则

只要图片链路没有严格经过：

`wechat-gen -> potato-illustrations -> ark-image-generation`

就视为整篇公众号包不合格。

此时必须：

- 停止生成
- 在审核记录中写明阻塞原因
- 不得输出“已完成”
- 不得伪造 `images/` 成品

## 本地交付目录

输出目录固定为：

```text
06_运营中心/<年份>/<YYYY-MM>/公众号/<YYYY-MM-DD>/<YYYYMMDD_公众号_<标题短名>/
```

必须包含：

```text
发布稿.md
草稿箱正文.txt
配图方案.md
正文结构与配图映射.md
封面方案.md
metadata.json
审核记录.md
prompts/
images/
```

## 结构规则

公众号稿不是一整篇平铺正文。

必须明确生成：

- 1 个一级标题
- 多个二级标题
- 必要的三级提示型小标题

不能整篇只有长段落。

## 配图规则

配图必须满足：

- 只解释一个核心判断
- 与对应段落一一绑定
- 明确写出插入到第几个二级标题下、第几段后
- 封面图不得混入正文图
- 正文图不得冒充封面图
- 每一张正式图都必须来自 `potato-illustrations`

## 封面规则

封面必须满足：

- 单独设计
- 默认交付横版 `2.35:1` 主封面
- 同时交付 `1:1` 方版封面
- 标题与副标题由本地叠字脚本完成，必须有清晰层级、美感和安全区控制
- `metadata.json.cover_path` 必须指向 `images/01_公众号封面_标题版_2.35比1.png`

## 禁止事项

出现任一项，直接打回：

- 未读取 `potato-illustrations` 就开始规划正式配图
- 在 `wechat-gen` 内直接调用生图脚本或 API
- 使用旧样张或旧通用图替代土豆崽图
- 用“先交文案、图后补”冒充完整公众号包
- `images/` 里没有正式图片却声称已完成
- 封面不是土豆崽体系
- 正文配图不是土豆崽体系
- 图片来源无法追溯到 `potato-illustrations`
- 如果图片有文字配图（不能有遮挡或重叠/穿模等问题）
- 如果图片有文字配图（文字字体，画风要美观，协调，不能强行弄个框框丢上去，整体和配图要协调）
- 如果图片文字没有经过本地叠字脚本的安全区、缩放、换行和层级检查
- 如果图片文字没有经过“安全区检查 -> 自动换布局 -> 必要时重生 raw -> 再失败则终止”的完整流程

## 完成定义

只有当下面全部成立时，才算完成：

1. 公众号正文结构已完成
2. 标题、摘要、CTA 已完成
3. 封面方案已完成
4. 正文配图方案已完成
5. `正文结构与配图映射.md` 已完成
6. 正式图片已由 `potato-illustrations` 生成并落到 `images/`
7. 审核记录已明确写出通过或打回

少任何一项，都不能叫完成。
